# ADR-0001：异构智能部落定位与交付顺序

- 状态：Accepted
- 日期：2026-07-10

## 背景

用户同时拥有 GPT、DeepSeek、Qwen、MiMo 等模型，但最强模型的 Token 和调用额度有限。现有多 Agent 产品通常强调工作流编排、通用 Agent 框架或更多并发调用，不能直接回答“在预算有限时，哪个成员应该做哪件事，以及这次选择是否比只用最强模型更划算”。

早期 Totemora 文档把 TUI、选举、议会、Web Observatory 和通用 multi-agent runtime 都列为核心，容易把项目推进成另一个编排框架，也无法验证部落的经济价值和成长是否真实。

## 决策

Totemora 定位为**预算约束下的异构智能组织系统**。

核心闭环固定为：

```text
任务分析 -> 预算约束 -> 可解释选人 -> 工作包执行 -> 独立验收
        -> 成本与质量对比 -> 经验提案 -> 审批后版本化成长
```

领域所有权如下：

- Totemora 自有：成员身份、能力画像、派工决策、运行评价、成长提案、信任与回滚。
- 外部协议可复用：模型 Provider、AG-UI、A2A、MCP、OpenTelemetry。
- 外部成熟产品不作为核心运行时依赖，避免领域模型被其抽象反向约束。

交付顺序为：

1. 固化当前 CLI、Provider、只读 Runtime 和 Trace 基线。
2. 建立 Run Schema、任务特征、选人理由、用量和失败归因。
3. 建立预算感知的可解释选人，并与单模型基线做重复实验。
4. 当经济与质量闭环成立后，接入可执行 Skill 和部落资产。
5. 有足够评价样本后，才实现可审批、可回滚的成员成长。
6. 最后建设 Web/AG-UI 控制面与更广泛生态接入。

## 方案比较

### 方案 A：继续做通用多 Agent 编排器

实现直观，但与现有框架同质化，无法形成预算与成员成长壁垒。不采用。

### 方案 B：先做 Web、市场和成员人格体验

展示效果强，但缺少派工质量和成本数据，成员“成长”容易退化为提示词堆积。不采用。

### 方案 C：先证明预算约束下的组织收益

产品面较窄，但能最早验证核心假设，并为 Skills、资产和成长提供可信数据。采用。

### 方案 D：暂不继续开发

可避免投入，但现有 Provider 和只读 Runtime 已足以支撑低成本实验，当前应先完成实验闭环而非停止。

## 后果

- “选了谁、为什么选、花了多少、谁验收、结果怎样”成为 Run 的必备证据。
- Chief 的自然语言派工只是早期候选方案，不能长期充当选人算法。
- 成员 profile 中的分数在有实测校准前只能视为先验值。
- Web、向量库和自动进化明确后置；Zvec 等资源先作为候选资产保留。
- 当前 Chief 同时终审是已知过渡状态，实验阶段需加入独立 Reviewer 或可重复的评价器。

## 验证

用 10 到 20 个可重复真实任务比较至少三种策略：单一强模型、固定廉价模型、Totemora 部落。记录验收通过率、强模型 Token、总 Token、估算费用、时延、重试次数和失败归因。若部落无法在成本或质量上形成稳定改进，应暂停 Skills、资产和成长功能，重新评估派工假设。
