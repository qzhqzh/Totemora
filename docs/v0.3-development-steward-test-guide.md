# v0.3 开发提交专员验收指南

## 前置条件

- 目标目录是 Git 仓库；
- 已配置本地 Git `user.name` 和 `user.email`；
- 当前没有 staged 文件；
- 当前没有 merge、rebase 或 cherry-pick；
- 工作树中存在准备提交的改动；
- DeepSeek、Qwen、MiMo Provider 可用。

## Web 验收

1. 启动 Gateway，打开 Web。
2. 从服务器 `.totemora/operator-token` 读取 Token，粘贴到页头“操作员 Token”。Token 只保存在该浏览器的 localStorage。
3. 登记目标 Git 仓库为 Workplace。
4. 选择该 Workplace，展开“安装 / 更新开发提交规范”。填写项目规范和允许的验证命令并保存。该动作只需首次或规范变化时执行。
5. 在任务大厅输入：`按这个项目的规范检查并提交当前改动`。
6. 确认 Task Analyzer 显示 `CHANGE · 已开放受控的现有改动提交`。
7. 等待 Chief、千工和米探生成 Proposal。
8. 核对文件、Commit message、风险、验证命令和 Reviewer 结论。
9. 点击“批准验证并提交”。
10. 确认所有验证通过并显示 40 位 Commit SHA。

## CLI 验收

CLI 默认连接 `http://127.0.0.1:4310`，并自动读取 `.totemora/operator-token`：

```bash
bun run totemora development prepare \
  --workplace <workplace_id> \
  --goal "按这个项目的规范检查并提交当前改动"
```

核对 Proposal 后批准：

```bash
bun run totemora development approve <proposal_id>
```

远程 Gateway 可指定：

```bash
TOTEMORA_OPERATOR_TOKEN=<token> \
bun run totemora development prepare \
  --gateway-url http://server:4310 \
  --workplace <workplace_id> \
  --goal "按规范提交当前改动"
```

## 必须验证的保护

- 新增 `.env` 后准备 Proposal，应在调用模型前拒绝。
- Proposal 生成后再修改任意批准文件，批准应失效。
- 验证命令失败时不得产生 Commit，且不得留下 staged 文件。
- 已有 staged 文件时必须拒绝准备 Proposal。
- Reviewer 拒绝时不能显示批准按钮。
- 工作流不得 push。

## 成长验证

第一次成功后，经验写入 `.totemora/member-experience/qwen_worker.json`。如果专员提出了通用改进，“开发提交规范”区域会显示待批准 Skill 提案。批准后版本写入 `.totemora/skills/git-change-management/active.json`。再次修改同一项目并准备 Proposal，千工会加载新 Skill 版本和最近少量已验证经验，而不是复用完整旧 Session。
