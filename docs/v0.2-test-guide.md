# Totemora v0.2 骨架验收指南

访问 `http://<服务器局域网 IP>:4310`，确认页头显示 `0.2.0-skeleton`。

## 1. 成功任务与独立验收

选择或登记 `examples/demo-project`，提交“分析订单折扣实现与 README 的差异”。保持最多成员 2、Run 总 Token 24000。

预期：显示候选排序、首领派工、成员能力匹配分、最终报告，以及由千工完成的独立 Reviewer 结论。任务队列最终为 completed。

## 2. 总 Token 硬预算

把 Run 总 Token 调到 1000 后提交同一任务。

预期：Runtime 在无法为下一次调用预留额度时停止；失败类别为 budget，不继续无限调用模型。

## 3. 成员数硬预算

把最多成员设为 1。任务若由 Chief 提议两名成员，Runtime 会要求修复方案；修复后仍超过预算则以 staffing 失败，不会绕过限制。

## 4. Mission 续接

成功执行一个任务后，在任务大厅选择已有 Mission，输入“继续检查此前遗漏的测试风险”。

预期：模式显示 CONTINUE；首领上下文包含历史请求和上次结果摘要；新 Run 归入同一 Mission。

## 5. Coding 门禁

输入“修复 pricing.ts 并运行测试”。

预期：识别为 CHANGE 或 OPERATE，显示当前未开放，不产生伪装成 Coding 的只读 Run。

## 6. 取消、重启与重试

任务运行中点击“取消 Run”，预期 Provider 请求被中止并记录 cancelled。对于 Provider、预算或派工等可重试失败，页面显示重试按钮。重启 Web 服务后，任务队列仍存在；重启时正在运行的 Job 会变为可重试的运行时中断。

## 当前非目标

v0.2 不执行文件修改、Shell、部署和数据库操作；不自动修改成员人格或能力。它验证的是长期驻扎、自然接活、预算派工证据、独立验收、失败治理和可恢复运行骨架。
